## إعداد بيئة التشغيل
- التحقق من تثبيت الأدوات: `dotnet --version`, `sqlcmd --version`, `flutter --version`.
- التأكد من توفر SQL Server محلياً أو تحديد سلسلة اتصال بديلة في `server/src/Hr.Api/appsettings.json`.

## تشغيل Web API
- استعادة وبناء المشروع: `dotnet restore` ثم `dotnet build server/src/Hr.Api/Hr.Api.csproj -c Release`.
- تشغيل الخدمة: `dotnet run --project server/src/Hr.Api/Hr.Api.csproj`.
- فحص جاهزية الخدمة عبر `GET http://localhost:5000/health`، وفتح Swagger: `http://localhost:5000/swagger`.

## معالجة الأخطاء أثناء التشغيل
- أخطاء بناء: عرض رسائل `dotnet build` ومعالجتها فوراً (نواقص حزم، مسارات مراجع المشاريع، أسماء النيمسبيس).
- أخطاء وقت التشغيل: فحص الاستثناءات في سجلّ التشغيل (الاتصال بقاعدة البيانات، JWT، CORS) وإصلاح الإعدادات في `Program.cs` و`appsettings.json`.
- اتصال قاعدة البيانات: إنشاء قاعدة `HrManagement` إذا لم تكن موجودة، إضافة مهاجرات عند الحاجة: `dotnet ef migrations add Init --project server/src/Hr.Infrastructure` ثم `dotnet ef database update --project server/src/Hr.Api`.

## اختبار API
- تشغيل اختبارات التكامل: `dotnet test server/tests/Hr.Api.Tests`.
- إضافة اختبارات وحدات/تكامل إضافية لمسارات الموظفين (CRUD، ترقيم صفحات، تصفية) حال ثبات التشغيل.

## ربط Flutter
- ضبط `ApiConfig.baseUrl` في التطبيق إلى عنوان الخدمة (مثل `http://localhost:5000/api/v1`).
- اختبار تسجيل الدخول: `POST /api/v1/auth/login` ثم إعداد التوكن في `ApiService.setAuthToken()`.
- اختبار جلب الموظفين عبر `EmployeesRepository.list()` ومشاهدة النتائج على الواجهة.

## أمان وجودة
- تمكين CORS مضبوطة للأصول المحلية.
- مراجعة سياسات الأدوار وربطها لاحقاً على المسارات.
- فحص الأداء الأولي: زمن استجابة `GET /employees`، وفهارس DB.

## التوثيق والمتابعة
- مراجعة Swagger لتوثيق جميع نقاط النهاية وإضافة أمثلة.
- تسجيل ما تم إصلاحه والتغييرات على الإعدادات/الكود.

## التسليم
- بعد نجاح التشغيل والاختبارات: تقديم تقرير بالأخطاء التي تم اكتشافها وإصلاحها، وتأكيد جاهزية الخدمة للتكامل مع بقية الوحدات.
